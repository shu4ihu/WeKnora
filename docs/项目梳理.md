# WeKnora 项目梳理

生成日期：2026-05-20

本文按照推荐阅读顺序逐项梳理 WeKnora 项目，目标是帮助新参与者快速建立项目全局视角：项目做什么、由哪些模块组成、请求如何进入系统、本地如何运行、AI/Trellis 开发流程如何配合。

## 0. 一句话定位

WeKnora 是一个企业级知识管理框架，围绕 RAG 问答、ReAct Agent 推理、MCP 工具调用和自动 Wiki 生成构建，支持多数据源接入、多模型厂商、多向量/检索后端和私有化部署。

## 1. 推荐阅读顺序总览

建议按以下顺序阅读：

1. `README_CN.md`：先理解项目定位、核心能力和产品边界。
2. `docs/开发指南.md`、`Makefile`：理解本地开发、运行和构建方式。
3. `cmd/server/main.go`、`internal/container/container.go`、`internal/router/router.go`：理解 Go 后端启动链路和请求入口。
4. `frontend/package.json`、`frontend/src/main.ts`、`frontend/src/router/index.ts`：理解前端技术栈、入口和页面结构。
5. `docreader/main.py`、`docreader/parser/parser.py`：理解文档解析服务如何被后端调用。
6. `mcp-server/README.md`、`mcp-server/main.py`：理解独立 MCP Server 如何暴露 WeKnora API 工具。
7. `client/README.md`、`cli/AGENTS.md`：理解 Go SDK 与 CLI 客户端。
8. `docker-compose.dev.yml`、`docker-compose.yml`、`helm/README.md`：理解开发环境和部署拓扑。
9. `AGENTS.md`、`.trellis/workflow.md`、`.trellis/spec/`：理解 AI 辅助开发流程和当前规范状态。

## 2. 第一步：读 `README_CN.md` 理解项目定位

### 2.1 文件作用

`README_CN.md` 是中文项目入口，适合先读。它回答三个核心问题：

- WeKnora 是什么。
- WeKnora 解决什么问题。
- WeKnora 当前有哪些核心能力。

### 2.2 关键结论

从 `README_CN.md` 可得出：

- WeKnora 面向企业级文档理解、语义检索和智能推理场景。
- 核心能力包括：
  - RAG 快速问答。
  - ReAct Agent 多步推理。
  - MCP 工具调用。
  - 网络搜索。
  - Wiki 模式，自动生成 Markdown 知识库和知识图谱。
- 支持外部数据源同步，例如飞书、Notion、语雀。
- 支持多类文档格式，例如 PDF、Word、图片、Excel 等。
- 支持多种模型供应商，例如 OpenAI、DeepSeek、Qwen、智谱、混元、Gemini、MiniMax、NVIDIA、Ollama、Claude 等。
- 支持私有化部署，组件可替换。
- 集成 Langfuse，用于 Agent、Token、任务流水线等观测。

### 2.3 建议关注点

读 README 时不要只看功能列表，建议重点建立这条产品主线：

```text
外部知识来源
  -> 文档解析
  -> 知识入库
  -> 分块 / 向量化 / 索引
  -> 检索 / 重排
  -> RAG 问答或 Agent 推理
  -> 前端、IM、MCP、CLI 等入口消费结果
```

这条链路会贯穿后端、DocReader、前端、MCP Server、CLI 和 SDK。

## 3. 第二步：读 `docs/开发指南.md` 和 `Makefile` 理解本地开发

### 3.1 `docs/开发指南.md` 的作用

`docs/开发指南.md` 给出推荐的本地开发方式。核心思想是：

- 基础设施用 Docker 启动。
- 后端 Go 服务本地运行。
- 前端 Vite 服务本地运行。
- 避免每次改代码都重新构建 Docker 镜像。

### 3.2 推荐本地开发流程

推荐流程是三步：

```bash
make dev-start
make dev-app
make dev-frontend
```

含义如下：

1. `make dev-start`
   - 启动开发依赖服务。
   - 包括 PostgreSQL、Redis、MinIO、Neo4j、DocReader、Jaeger 等。

2. `make dev-app`
   - 在本地直接运行 Go 后端。
   - 修改后端代码后通常需要停止并重启。

3. `make dev-frontend`
   - 启动前端 Vite 开发服务器。
   - 支持热更新。

常见访问地址：

```text
前端开发服务器: http://localhost:5173
后端 API:        http://localhost:8080
PostgreSQL:      localhost:5432
Redis:           localhost:6379
MinIO Console:   http://localhost:9001
Neo4j Browser:   http://localhost:7474
Jaeger UI:       http://localhost:16686
```

### 3.3 `Makefile` 的作用

`Makefile` 是项目的主要命令入口。常用目标包括：

```bash
make build             # 构建 Go 后端
make run               # 构建并运行 Go 后端
make test              # 运行 Go 测试
make dev-start         # 启动开发环境基础设施
make dev-stop          # 停止开发环境
make dev-status        # 查看开发环境状态
make dev-app           # 本地运行后端
make dev-frontend      # 本地运行前端
make docker-build-all  # 构建全部镜像
make build-lite        # 构建 Lite 版本
make run-lite          # 运行 Lite 版本
```

从 `Makefile` 可确认：

- Go 后端二进制名是 `WeKnora`。
- 后端主入口是 `./cmd/server`。
- Docker 镜像包括 app、docreader、frontend。
- 数据库迁移通过 `scripts/migrate.sh` 执行。
- Lite 模式会把前端构建进 Go 服务中，面向零外部依赖或桌面场景。

## 4. 第三步：读后端启动链路

建议按下面三个文件顺序阅读：

1. `cmd/server/main.go`
2. `internal/container/container.go`
3. `internal/router/router.go`

### 4.1 `cmd/server/main.go`：进程入口

`cmd/server/main.go` 是 Go 后端服务入口。主要职责：

- 根据 `GIN_MODE` 设置 Gin 运行模式。
- 打印启动环境信息。
- 构建依赖注入容器。
- 从容器中取出配置、Router、Tracer、资源清理器。
- 创建 HTTP Server。
- 监听配置中的 `host:port`。
- 注册优雅退出逻辑。
- 退出时清理资源。

简化链路：

```text
main()
  -> runtime.LogStartupEnv
  -> container.BuildContainer
  -> c.Invoke(...)
  -> http.Server{Handler: router}
  -> listenWithRetry
  -> server.Serve
  -> shutdown signal
  -> resourceCleaner.Cleanup
```

这个文件适合用来理解“服务是怎么启动起来的”，但业务逻辑不在这里。

### 4.2 `internal/container/container.go`：依赖装配中心

`internal/container/container.go` 是整个后端的依赖注入装配中心，使用 `dig.Container` 注册项目组件。

它把系统拆成几类依赖：

1. 核心基础设施
   - 配置加载。
   - Tracer。
   - Langfuse。
   - 数据库。
   - 文件服务。
   - Redis。
   - goroutine pool。

2. 外部服务客户端
   - DocReader gRPC Client。
   - Ollama。
   - Neo4j。
   - DuckDB。
   - Stream Manager。

3. Repository 层
   - Tenant、KnowledgeBase、Knowledge、Chunk、Session、Message、Model、User 等。
   - WikiPage、WikiLogEntry、TaskPendingOps、TaskDeadLetter 等。
   - 向量/检索相关 Repository，例如 Qdrant、Milvus、Weaviate、Doris、Elasticsearch、Postgres、Tencent VectorDB 等。

4. Service 层
   - TenantService。
   - KnowledgeBaseService。
   - KnowledgeService。
   - ChunkService。
   - ModelService。
   - SessionService。
   - AgentService。
   - WebSearchService。
   - VectorStoreService。
   - WikiPageService。
   - WikiIngestService。

5. Handler 层和 Router
   - 注册 HTTP Handler。
   - 注册最终 Gin Router。

6. 异步任务
   - 如果配置了 Redis，则注册 Asynq。
   - 如果没有 Redis，则降级为同步任务执行器。

阅读这个文件的价值是：可以看到 WeKnora 后端有哪些能力，以及这些能力之间大致如何装配。

### 4.3 `internal/router/router.go`：HTTP 请求入口

`internal/router/router.go` 创建 Gin Router，并注册中间件和 API 路由。

启动时主要做：

1. 创建 `gin.New()`。
2. 注册 CORS。
3. 注册基础中间件：
   - RequestID
   - Language
   - Logger
   - Recovery
   - ErrorHandler
4. 注册 `/health`。
5. 非 release 模式下注册 Swagger：`/swagger/*any`。
6. Lite 版本注册前端静态文件。
7. 认证前注册 IM 回调路由。
8. 注册 Auth 中间件。
9. 注册文件服务、预签名文件访问、Langfuse、AuditLog 等中间件。
10. 注册 `/api/v1` 下的业务路由。

`/api/v1` 下的重要路由模块包括：

- Auth
- Tenant
- KnowledgeBase
- KnowledgeTag
- Knowledge
- FAQ
- Chunk
- Session
- Chat
- Message
- Model
- Evaluation
- System
- MCP Service
- Web Search
- Vector Store
- Custom Agent
- Skill
- Organization
- IM Channel
- Data Source
- WeKnora Cloud
- Wiki Page
- Chunker Debug

简化请求链路：

```text
HTTP request
  -> Gin middleware
  -> Auth / RBAC / audit / langfuse
  -> /api/v1 route
  -> handler
  -> service
  -> repository
  -> database / vector store / object storage / external service
```

### 4.4 后端目录理解方式

后端主要代码在 `internal/`。建议按职责理解，而不是一次性读完：

```text

internal/application/repository/ 数据访问与检索实现
internal/config/            配置结构与加载
internal/middleware/        Gin 中间件
internal/models/            模型供应商与 embedding 等能力
internal/mcp/               后端 MCP 相关能力
internal/im/                IM 平台适配
internal/datasource/        外部数据源连接器
internal/tracing/           链路追踪与 Langfuse
```

## 6. 第五步：读 DocReader 文档解析服务

建议读：

1. `docreader/main.py`
2. `docreader/parser/parser.py`
3. `docreader/parser/` 下具体 parser

### 6.1 DocReader 的职责

DocReader 是独立 Python gRPC 服务，负责把文件或 URL 解析为 Markdown 内容，并把文档中的图片引用返回给 Go 后端处理。

它不负责：

- 分块。
- 存储。
- OCR 总调度。
- 向量化。
- 知识入库。

这些职责留给 Go App 或其他模块。

### 6.2 `docreader/main.py`：gRPC 服务入口

主要结构：

```text
DocReaderServicer
  -> Read(request, context)
  -> ListEngines(request, context)

main()
  -> config.print_config
  -> AuthInterceptor
  -> grpc.server
  -> add_DocReaderServicer_to_server
  -> add HealthServicer
  -> load TLS credentials
  -> start server
```

`Read` 支持两种模式：

1. URL 模式：`request.url` 有值。
2. File 模式：`request.file_content` 有值。

解析后返回：

- `markdown_content`
- `image_refs`
- `image_dir_path`
- `metadata`
- `error`

图片处理逻辑中，DocReader 会把图片解码成 inline bytes，通过 `ImageRef` 返回；实际持久化由 Go App 负责。

### 6.3 `docreader/parser/parser.py`：解析门面

`Parser` 是轻量门面：

- `parse_file`：根据文件类型和 parser engine 从 registry 中选择具体 parser。
- `parse_url`：用 `WebParser` 解析 URL。

核心链路：

```text
DocReaderServicer.Read
  -> Parser.parse_file / Parser.parse_url
  -> registry.get_parser_class
  -> concrete parser.parse
  -> Document(content, images, metadata)
  -> ReadResponse
```

这说明 DocReader 是可扩展的：新增解析引擎或文件类型时，重点看 `docreader/parser/registry.py` 和具体 parser 实现。

## 8. 第七步：读 Go Client SDK 与 CLI

### 8.1 `client/README.md`：Go HTTP Client SDK

`client/` 是 Go HTTP 客户端库，目标是让其他模块或外部程序不用直接写 HTTP 请求即可调用 WeKnora API。

它覆盖的能力包括：

- 会话管理。
- 知识库管理。
- 知识管理。
- 租户管理。
- 知识问答。
- Agent 问答。
- 分块管理。
- 消息管理。
- 模型管理。

典型使用方式：

```go
apiClient := client.NewClient(
    "http://api.example.com",
    client.WithToken("your-auth-token"),
    client.WithTimeout(30*time.Second),
)
```

支持通过 `WithTenantID` 设置默认租户，也支持从 `context` 临时切换租户。

阅读 SDK 时建议按 API 领域查找：

```text
client/session*
client/knowledge*
client/knowledge_base*
client/model*
client/agent*
client/chunk*
```

实际文件名可能需要按当前目录再确认。

### 8.2 `cli/AGENTS.md`：CLI 开发规范

`cli/` 是 Go 写的 `weknora` 命令行客户端。`cli/AGENTS.md` 是该子项目最重要的开发规范。

核心架构：

```text
cmd/main.go
  -> cmd.Execute()
  -> cmd.NewRootCmd(cmdutil.New())
```

关键包：

```text
cmd/<name>/             cobra command 实现
internal/cmdutil/       Factory、FormatOptions、typed Error、exit code、确认协议等
internal/format/        JSON 输出
internal/iostreams/     IO 和 TTY 检测
internal/secrets/       token / secret 存储
internal/prompt/        交互提示
internal/sse/           SSE 累积器
internal/mcp/           CLI 内置 MCP server
client/                 父模块 SDK
```

CLI 命令遵循 Options + Narrow Service Pattern：

1. 每个命令定义自己的 `Options`。
2. 每个命令定义自己的窄 `Service` interface。
3. `NewCmd<Verb>` 负责 flag 注册。
4. `run<Verb>` 负责业务逻辑，也是测试注入点。

重要规范：

- 不要跨命令共享 Service interface。
- 不要在构造命令时初始化 client，否则 `--help` 也会触发认证。
- JSON 输出必须通过 `cmdutil.AddFormatFlag` 和 `FormatOptions.Emit`。
- 删除、清空、覆盖等破坏性写操作必须走确认协议。
- SDK 调用错误要用 `cmdutil.WrapHTTP` 等 typed error 包装。
- 测试使用窄 fake，不用 mocking library。
- 提交前至少通过：

```bash
go test -count=1 ./...
go vet ./...
```

CLI 是当前项目里规范最完整的子模块。如果要继续完善 `.trellis/spec/backend/`，`cli/AGENTS.md` 是很好的来源。

### 9.3 `helm/README.md`：Kubernetes 部署

Helm Chart 面向 Kubernetes 部署。架构如下：

```text
Ingress
  -> Frontend (Vue.js)
  -> Backend (Go/Gin)
       -> DocReader (gRPC)
       -> PostgreSQL (ParadeDB)
       -> Redis (Queue)
```

基础要求：

- Kubernetes 1.25+

### 10.1 `AGENTS.md`：项目级 AI 入口说明

### 10.2 `.trellis/workflow.md`：任务流程

当前项目的 Trellis 流程大意是：

1. 无任务时：
   - 纯问答、解释、少量只读查找可以直接回答。
   - 代码变更、重构、构建类工作要创建任务。

2. Plan 阶段：
   - 创建任务。
   - 用 brainstorm 明确需求。
   - 写 PRD。
   - 配置实现和检查上下文。

3. Execute 阶段：
   - 实现。
   - 检查。
   - 质量验证。

4. Finish 阶段：
   - 确认变更完成。
   - 必要时更新 spec。
   - 提交后结束任务。

## 12. 常见开发切入点

### 12.1 改后端 API

优先阅读：

```text
internal/router/router.go
internal/handler/<domain>.go
internal/application/service/<domain>.go
internal/application/repository/<domain>.go
internal/types/interfaces/
```

判断是否需要同步修改：

- 数据库迁移。
- 前端 API 调用。
- Client SDK。
- CLI。
- Swagger / docs。
- 权限和 RBAC。

### 12.2 改前端页面

优先阅读：

```text
frontend/src/router/index.ts
frontend/src/views/<domain>/
frontend/src/api/
frontend/src/stores/
frontend/src/components/
```

改 UI 后应本地启动并浏览器验证：

```bash
make dev-start

make dev-frontend
```

### 12.3 改文档解析

优先阅读：

```text
docreader/main.py
docreader/parser/parser.py
docreader/parser/registry.py
docreader/parser/<specific_parser>.py
internal/infrastructure/docparser/
```

注意职责边界：DocReader 返回 Markdown、图片引用和 metadata；存储、入库、分块通常在 Go App 侧。

### 12.4 改 CLI

优先阅读：

```text

cli/cmd/<domain>/
cli/internal/cmdutil/
cli/internal/format/
cli/internal/iostreams/
```

必须遵循 CLI 的 JSON 输出、错误码、确认协议和测试规范。

### 12.5 改部署

优先阅读：

```text
Makefile

docker-compose.yml
docker/
helm/README.md
helm/values.yaml

```

部署改动通常影响较大，需要注意 Secret、持久化、健康检查、端口、profile、镜像 tag 和生产环境兼容性。

## 13. 推荐的下一步深入专题

如果继续深入，建议按专题拆：

1. 后端知识入库链路
   - 从上传文件 API 开始，追踪 Handler -> Service -> DocReader -> Chunk -> Embedding -> VectorStore。

2. 会话与 RAG 问答链路
   - 从 `/chat` 或 Session 相关路由开始，追踪检索、重排、LLM 调用、流式返回。

3. Agent 执行链路
   - 关注 AgentService、MCP、WebSearch、工具调用、Langfuse tracing。

4. Wiki 入库链路
   - 关注 WikiIngest、任务队列、死信队列、WikiPage、图谱关系。

5. 前端知识库页面
   - 从 `frontend/src/router/index.ts` 的 knowledge-bases 路由进入页面和 API。

6. CLI 子项目
   - 按 `cli/AGENTS.md` 梳理命令结构、错误码、JSON wire contract 和内置 MCP 工具面。
